# REGLAS DE VALIDACIÓN - CARTA DE CERTIFICACIÓN PRAGMA S.A.

Este conjunto de reglas define los criterios objetivos para la validación de documentos corporativos bajo el estándar de PRAGMA S.A.

---

## 🏗️ 1. REGLAS DE ESTRUCTURA (OBLIGATORIAS)

* **R-EST-01 (Título):** El documento debe iniciar con el encabezado exacto: "Carta de certificación {Nombre del Proyecto}".
* **R-EST-02 (Fecha):** Debe seguir el formato: `Bogotá, {día} de {mes en español} de {año}`.
* **R-EST-03 (Destinatario):** Bloque estructurado que incluya: Señores, Empresa, Atn (nombre), Cargo y Ciudad.
* **R-EST-04 (Asunto):** Debe incluir el nombre específico del producto o servicio certificado.
* **R-EST-05 (Bloque de Firma):** Debe contener obligatoriamente: Nombre del analista, Cargo y el nombre "PRAGMA S.A.".

---

## 📝 2. REGLAS DE CONTENIDO TÉCNICO

* **R-CON-01 (Identificación):** Mencionar explícitamente el Nombre del proyecto y la Versión certificada.
* **R-CON-02 (Alcance Técnico):** Declarar los Ambientes de ejecución y los Tipos de pruebas realizadas (ej. manuales, performance).
* **R-CON-03 (Dictamen):** Declarar explícitamente el cumplimiento de requisitos y la aptitud para producción o integración.
* **R-CON-04 (Listado de Entregables):** Identificar los elementos certificados (HUs, módulos, etc.).
* **R-CON-05 (Observaciones):** Contener una sección separada denominada "Observaciones".

---

## 🎨 3. REGLAS DE FORMATO Y ESTILO

* **R-FOR-01 (Tipografía Word):** Fuente Arial, Tamaño 12, Interlineado 1.5 (Aplica solo para archivos .docx).
* **R-FOR-02 (Viñetas):** El listado de elementos debe usar viñetas (bullets), no numeración.
* **R-FOR-03 (Idioma):** No se permite el uso de meses en inglés; todo el contenido debe estar en español.
* **R-FOR-04 (Segmentación):** Separación clara y visual entre secciones del documento.

---

## ✍️ 4. REGLAS LINGÜÍSTICAS Y DE CALIDAD

* **R-LIN-01 (Ortografía):** Cero tolerancia a errores ortográficos, tipográficos o de tildes.
* **R-LIN-02 (Terminología):** Se permite el uso de tecnicismos en inglés (ej. "performance") siempre que sean correctos en el contexto de QA.
* **R-LIN-03 (Claridad):** La redacción no debe ser ambigua ni afectar la interpretación técnica.
